## 1. 产品愿景

### 1.1 什么是 Vibe Coding？

**Vibe Coding** = **氛围编程**：在轻松的氛围中，通过自然对话创造软件。

核心理念：
- **不是写代码，而是描述想法** (Not coding, but describing)
- **不是技术对话，而是日常交流** (Not technical, but conversational)
- **不是独自工作，而是与AI伙伴协作** (Not alone, but with AI partner)

### 1.2 为什么选择移动端？

| 市场 | 现状 | 机会 |
|------|------|------|
| **桌面端** | Bolt.new, v0, Lovable 等已成熟 | 竞争激烈 |
| **移动端** | 几乎空白 | 蓝海市场 |

**移动端优势**：
- ✅ 随时随地记录想法
- ✅ 触摸交互更直观
- ✅ 语音输入更自然
- ✅ 非技术用户更习惯手机操作

### 1.3 产品愿景

> **让每个人在手机上通过对话创造软件，就像和朋友聊天一样自然。**

## 2. 目标用户

### 2.1 核心用户画像

#### 用户 A: 小企业主 - 李姐 (45岁)
- **需求**: 快速做一个预约管理页面
- **技术能力**: 会用微信，不会编程
- **场景**: 在咖啡馆用手机与AI对话，10分钟生成预约页面

#### 用户 B: 内容创作者 - Alex (26岁)
- **需求**: 快速做个人作品集落地页
- **技术能力**: 熟悉社交媒体，基础HTML懂一点
- **场景**: 在地铁上用手机描述需求，生成可分享的链接

- **需求**: 快速做活动报名页面
- **技术能力**: 会用办公软件，不懂编程

## 3. 核心价值主张

### 3.1 对话式创造

```
传统方式:
用户 → 学习编程 → 写代码 → 调试 → 完成
   (数周到数月)

Vibe Coding:
用户 → 与AI对话 → 预览 → 微调 → 完成
   (10-30分钟)
```

### 3.2 移动端优先

- **语音输入**: "帮我做一个活动报名页面"
- **触觉反馈**: 点赞/调整来引导AI
- **实时预览**: 右侧预览，左侧对话
- **手势操作**: 滑动切换页面，长按查看代码

### 3.3 中文原生

- 理解中文表达习惯
- 支持语音输入
- 中文化的设计系统

## 4. 与竞品的差异化

### 4.1 竞品对比

| 产品 | 目标用户 | 使用场景 | 移动端 | 中文 | Vibe Coding |
|------|----------|----------|--------|------|-------------|
| **v0.dev** | 开发者 | 生成UI组件 | ❌ | ❌ | ❌ |
| **Bolt.new** | 技术用户 | 全栈开发 | ⚠️ | ⚠️ | ⚠️ |
| **Lovable** | 创业者 | 快速MVP | ❌ | ⚠️ | ⚠️ |
| **Replit** | 学习者 | 编程学习 | ❌ | ⚠️ | ❌ |
| **Zaoya** | **非技术用户** | **移动端创造** | **✅** | **✅** | **✅** |

### 4.2 差异化策略

#### 差异点 1: 移动端原生体验
```
竞品: 桌面端为主，移动端勉强可用
Zaoya: 从第一天就为移动端设计
  - 大按钮，易点击
  - 语音输入优先
  - 单手操作友好
  - 实时预览 + 对话分屏
```

#### 差异点 2: 对话式交互
```
竞品: 技术对话（"写一个React组件"）
Zaoya: 自然对话（"我想做个活动报名"）
  - Interview Agent 主动提问澄清需求
  - Emoji 反馈增强亲和感
  - 日常语言，无需技术术语
```

#### 差异点 3: 引导式创作
```
竞品: 自由输入，用户需要知道怎么问
Zaoya: 结构化引导
  - 4 大模板类别
  - Interview Agent 自适应提问
  - 快捷操作芯片 (Quick Action Chips)
```

  ↓

用户: "帮我做一个活动报名页面"

AI: "好的！为了帮你做出最合适的页面，我想了解几个细节：

## 9. 关键设计决策

### 9.1 为什么选择移动端优先？

| 决策 | 理由 |
|------|------|
| **移动端优先** | 蓝海市场，竞品少 |
| **桌面端兼容** | 后续支持桌面端浏览器访问 |
| **渐进增强** | 从简单到复杂逐步进化 |

### 9.2 为什么选择 React Native？

| 技术 | 优势 |
|------|------|
| **React Native** | 跨平台 (iOS + Android) |
| **Web 原生** | 预览使用 WebView，与最终输出一致 |
| **生态成熟** | 第三方库丰富 |

### 9.3 为什么选择多 Agent 架构？

参考 **Kimi CLI** 的 LaborMarket 模式:

| Agent | 职责 | 好处 |
|-------|------|------|
| **Interview Agent** | 需求澄清 | 上下文更清晰 |
| **Generation Agent** | 生成页面 | 专注生成质量 |
| **Refinement Agent** | 优化迭代 | 基于反馈快速调整 |

**优势**:
- ✅ 每个 Agent 专注一件事
- ✅ 上下文更干净
- ✅ 易于调试和优化
- ✅ 符合单一职责原则

### 9.4 为什么选择模板 + Interview 混合模式？

| 纯模板 | 纯对话 | **模板 + Interview** |
|--------|--------|---------------------|
| 灵活性差 | 不可控 | ✅ 平衡灵活性和可控性 |
| 快速但死板 | 慢且易错 | ✅ 既有速度又有质量 |
| 无需澄清 | 需大量澄清 | ✅ 减少澄清轮次 |

### 9.5 为什么选择 SSE (Server-Sent Events)？

| 技术 | 适用场景 | 选择 |
|------|----------|------|
| **轮询** | 简单但延迟高 | ❌ |
| **WebSocket** | 双向实时 | ⚠️ 过度设计 |
| **SSE** | 单向推送 | ✅ **最适合** |

**选择 SSE 的理由**:
- ✅ 单向推送 (服务器 → 客户端)
- ✅ 自动重连
- ✅ 浏览器原生支持
- ✅ 比 WebSocket 简单
